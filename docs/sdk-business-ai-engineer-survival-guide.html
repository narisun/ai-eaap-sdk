YOUR CHARTER

Build agents that solve real business problems.

You're hired to bring domain knowledge to AI — to translate the messy, specific reality of your business into prompts, tools, schemas, and evaluations that move metrics that matter. That's the work only you can do.

You don't have to learn OPA, OTel, LangGraph internals, MCP transports, or how retries interact with timeouts. The platform team handles all of that, behind clean interfaces. You hold them to that contract; they hold you to writing agents the SDK can safely run.

What you own

  • Prompts & system messages
  • Tool design — what's a tool, at what granularity
  • Pydantic input/output schemas for tools
  • Choice of model (gpt-4o-mini, sonnet, etc.)
  • RAG pipelines & retrieval quality
  • Eval harnesses against business outcomes
  • Agent topology — single, hand-off, supervisor
  • Deciding which OPA opa_path a tool needs

What the platform team owns

  • JWT verification & OPA wiring
  • Audit pipeline & redactors
  • OpenTelemetry & LangFuse integration
  • Budget enforcement & daily limits
  • Retry, timeout, and cancellation policies
  • MCP connection pool & lifecycle
  • Compaction internals (you configure thresholds)
  • Test fakes for every interface

CH 01

Day one.

If a hello-world agent answers a question by lunch, you've cleared the highest hurdle. Everything after this is iteration.

$ shellinstall + scaffold# 1. install the SDK
pip install ai-core-sdk

# 2. scaffold a project
eaap new my-agents
cd my-agents

# 3. run the example agent
python -m my_agents.main
my_agents/main.py~30 lines · the whole hello-worldfrom ai_core import AICoreApp, BaseAgent, tool
from pydantic import BaseModel

class HiIn(BaseModel):  name: str
class HiOut(BaseModel): greeting: str

@tool(name="say_hi", version=1)
async def say_hi(p: HiIn) -> HiOut:
    return HiOut(greeting=f"Hi {p.name}!")

class Greeter(BaseAgent):
    agent_id = "greeter"
    def system_prompt(self) -> str:
        return "You greet people warmly and end with a smiley."
    def tools(self):
        return [say_hi]

async def main() -> None:
    async with AICoreApp() as app:
        agent = app.agent(Greeter)
        out = await agent.invoke({"messages": [
            {"role": "user", "content": "hi, I'm Sam"}
        ]})
        print(out["messages"][-1])

if __name__ == "__main__":
    import asyncio; asyncio.run(main())
tipThat's the whole agent. The SDK supplied the other ~10,000 lines of cross-cutting infrastructure you didn't have to write. Reread the file and notice what's not there: no auth, no retry, no audit, no observability code. All of that is wired automatically by AICoreApp().
watch outThe dev defaults bind a no-op policy evaluator and an env-based secret manager. That's fine for local dev. Production deployments use ProductionSecurityModule — see CH 09.
CH 02

Anatomy of an agent.

The smallest mental model that's accurate. BaseAgent compiles to a LangGraph; you don't usually need to know that, but the seams are useful when you do.

You override three methods: system_prompt(), tools(), mcp_servers(). Optionally extend_graph() if you need custom nodes. The SDK builds the rest of the LangGraph for you and wraps it in observability, audit, and budget enforcement.

key ideaAn agent is data + functions: the system prompt (data), the tools (functions). Most of your design work is deciding what's a tool and what its schema looks like. CH 03.
CH 03

Writing tools.

A tool is an async def with a single Pydantic input and a Pydantic output, decorated with @tool. Three pieces. That's the whole contract.

tools/lookup_ticket.pythe canonical shapefrom ai_core.tools import tool
from pydantic import BaseModel, Field

class LookupTicketIn(BaseModel):
    ticket_id: str = Field(..., min_length=1, pattern=r"^T-\d+$")

class LookupTicketOut(BaseModel):
    ticket_id: str
    status:    str
    summary:   str

@tool(
    name="lookup_ticket",
    version=1,
    description="Fetch a ticket by id.",
    opa_path="eaap/support/lookup",
)
async def lookup_ticket(p: LookupTicketIn) -> LookupTicketOut:
    # your business logic, full stop.
    row = await tickets.fetch(p.ticket_id)
    return LookupTicketOut(**row)
three rules1. Tools are async def, never sync. 2. Single Pydantic positional argument. Pydantic return type. 3. Module-level functions, not methods or classmethods.
opa pathEvery tool needs an opa_path. The platform team writes the policy at that path; you decide what scope makes sense. Use namespaces like eaap/<area>/<action>. Pass opa_path=None only for tools so generic they need no policy (and even then, run it past the platform team).

Errors that propagate from tools

errors you'll seeand what they meanfrom ai_core import (
    ToolValidationError,    # bad input — pydantic parse failed
    PolicyDenialError,      # OPA said no
    ToolExecutionError,     # your handler raised something unexpected
    SchemaValidationError,  # your output doesn't match the schema
    BudgetExceededError,    # this tool's LLM call would blow the daily limit
)

Catch specific exceptions. Don't swallow with except Exception — you'll mask budget and policy errors and look like you're handling them when you aren't.

CH 04

Calling LLMs.

Inside BaseAgent the LLM is already wired up — you set the system prompt and tool list, the SDK does the rest. When you need to call an LLM from inside a tool, do it through the bound ILLMClient.

tools/summarise.pyresolve via the containerfrom ai_core.di.interfaces import ILLMClient
from ai_core.tools import tool

class SummariseIn(BaseModel):  text: str; max_tokens: int = 200
class SummariseOut(BaseModel): summary: str

@tool(name="summarise", version=1)
async def summarise(p: SummariseIn) -> SummariseOut:
    llm: ILLMClient = current_container().get(ILLMClient)
    resp = await llm.complete(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "Summarise concisely."},
            {"role": "user",   "content": p.text},
        ],
        max_tokens=p.max_tokens,
        tenant_id=current_tenant(),     # required for budget + obs
        agent_id=current_agent(),       # required for budget + obs
        temperature=0.0,
    )
    return SummariseOut(summary=resp.content.strip())
always passtenant_id and agent_id on every llm.complete() call. Without them, budget enforcement quietly doesn't apply and observability loses its join keys. The first time you find out is on the AWS bill.

Switching models

Change EAAP_LLM__DEFAULT_MODEL to switch the SDK's default. Override per-call by passing model="...". LiteLLM handles the protocol differences — the same code talks to OpenAI, Anthropic, Vertex, Bedrock, etc.

model pickingStart with gpt-4o-mini or claude-haiku for cost. Move to a stronger model only when an eval shows it's needed. Cost rises faster than capability for most business tasks.
CH 05

Memory & compaction.

Long conversations exceed model context. The SDK's MemoryManager compacts history into a summary while preserving Essential Entities. You configure thresholds in env vars; the rest is automatic.

.envconfiguration only# Above this many tokens, run compaction
EAAP_AGENT__MEMORY_COMPACTION_TOKEN_THRESHOLD=12000

# Target post-compaction size — passed into the summariser prompt
EAAP_AGENT__MEMORY_COMPACTION_TARGET_TOKENS=2000

# Hard ceiling on the summariser LLM call
EAAP_AGENT__COMPACTION_TIMEOUT_SECONDS=30

# Keys whose values must survive compaction verbatim
EAAP_AGENT__ESSENTIAL_ENTITY_KEYS=customer_id,ticket_id,policy_number
when neededSingle-turn agents (lookup-and-respond) don't need compaction — there's nothing to compact. Multi-turn agents (long support sessions, research loops) need it. Default settings work for most cases; tune only when an eval surfaces a problem.
essential entitiesAdd anything that should survive compaction — customer ids, ticket numbers, policy references, decision IDs. The summariser receives them as a list and is instructed to preserve them verbatim.
CH 06

MCP servers.

An MCP server is a separately-running process that exposes tools, resources, or prompts via the Model Context Protocol. Use one when you have a substantial body of capability to share across agents — your own KB, a vendor's tools, an existing internal service.

agents/research_agent.pydeclare, agent picks up toolsfrom ai_core.mcp import MCPServerSpec
from ai_core import BaseAgent

class ResearchAgent(BaseAgent):
    agent_id = "research"

    def system_prompt(self) -> str:
        return "You research and cite sources."

    def mcp_servers(self):
        return [
            MCPServerSpec(
                component_id="web-search",
                transport="http",
                url="http://web-search.internal:8080",
            ),
            MCPServerSpec(
                component_id="kb",
                transport="stdio",
                command=["python", "-m", "kb_mcp.server"],
            ),
        ]
tool vs MCPUse @tool for a single function or a small cohesive group. Use an MCPServerSpec when the capability is large enough to deserve its own process — your own knowledge base, a vendor's full toolkit, an existing internal service you're wrapping.
tool name collisionsIf two MCP servers expose tools with the same name, the SDK raises rather than silently picking one. Either rename upstream or scope through component_id.
CH 07

Tests without external services.

A unit test of an agent should run in 0.04 seconds and need zero external services. The SDK provides fakes for every interface; you wire them up with Container.override().

tests/test_lookup_ticket.pyfake everythingfrom ai_core.di import Container
from ai_core.di.module import AgentModule
from ai_core.testing import (
    FakeModule, FakePolicyEvaluator, FakeBudgetService, FakeObservabilityProvider
)
from ai_core.di.interfaces import IPolicyEvaluator
import pytest

@pytest.fixture
def container():
    c = Container.build([AgentModule(), FakeModule()])
    c.override(IPolicyEvaluator, FakePolicyEvaluator(default_allow=True))
    return c

async def test_lookup_ticket_allowed(container):
    invoker = container.get(ToolInvoker)
    out = await invoker.invoke(
        lookup_ticket,
        arguments={"ticket_id": "T-42"},
        agent_id="customer-support",
        tenant_id="acme",
    )
    assert out.status == "open"
the test seamContainer.override(I, fake) is the only test seam you need. Don't monkey-patch SDK internals — the bindings drift across versions and tests break invisibly. Fakes don't.
contractIf a test fails with "PostgreSQL connection refused" or "OPA unreachable", you forgot a fake. Build the container with FakeModule() and assert: this is a unit test, not an integration test.
CH 08

Evaluating agents.

Latency and token count are infrastructure metrics. Business outcomes are the metrics that matter for an agent. Resolution rate, escalation rate, cost-per-task, customer satisfaction. If you can't measure it, the agent isn't working — you just don't know yet.

A useful eval has three layers

  1. L1 Golden cases. A handful of hand-curated examples with known correct answers. Run on every PR. Catches obvious regressions.
  2. L2 LLM-as-judge sweep. Larger set of cases where a strong model rates the agent's output against a rubric. Run nightly. Catches drift.
  3. L3 Production telemetry. Aggregate business metrics — resolution rate, deflection rate, escalation rate. Run continuously. Catches everything else.
eval before improvementIf you don't have an eval, you can't tell whether a "better prompt" is actually better. Build the eval first; iterate on the agent second.
CH 09

Deploying agents.

Most production agents go behind FastAPI with JWT + OPA enforcement. The SDK provides OPAAuthorization as a drop-in dependency.

api/app.pyJWT + OPA on every routefrom fastapi import FastAPI, Depends
from ai_core.di import Container
from ai_core.di.module import AgentModule, ProductionSecurityModule
from ai_core.security import OPAAuthorization, AuthorizedPrincipal
from ai_core.observability.logging import bind_context, reset_context

container = Container.build([AgentModule(), ProductionSecurityModule()])
authz = OPAAuthorization(container, decision_path="eaap/api/allow")
app = FastAPI()

@app.post("/agents/support/run")
async def run_support(
    req: RunRequest,
    principal: AuthorizedPrincipal = Depends(
        authz.requires(action="agent.run", resource="customer-support")
    ),
):
    tokens = bind_context(
        tenant_id=principal.claims["tenant_id"],
        user_id=principal.subject,
        request_id=req.request_id,
    )
    try:
        return await support_agent.invoke(req.payload)
    finally:
        reset_context(tokens)
production moduleAgentModule() alone gives you a no-op policy evaluator — fine for dev, not for prod. Use AgentModule() + ProductionSecurityModule() for any environment that holds real customer data.
ask the platform teamYour team probably has a deploy template (k8s manifests, env-var bundles) — get it on day one rather than rolling your own. The platform team has standardised env vars, health probes, and graceful-shutdown timing.
CH 10

Cookbook recipes.

The handful of tasks that come up most weeks. Each is a numbered sequence — copy and adapt.

Add a tool to an existing agent

  1. Create tools/your_tool.py with a Pydantic input/output and an async def handler.
  2. Decorate with @tool(name=..., version=1, opa_path=...).
  3. Add the import to the agent's tools() return list.
  4. Write a unit test using FakeModule + FakePolicyEvaluator(default_allow=True).
  5. Coordinate with the platform team to add the OPA rule under your opa_path in the policy bundle.

Switch from GPT-4o-mini to Claude Sonnet

  1. Update EAAP_LLM__DEFAULT_MODEL=anthropic/claude-sonnet-4-5 in your settings.
  2. Set the credential (ANTHROPIC_API_KEY) — the platform team handles this in production.
  3. Re-run your evals — temperature and max-tokens may need re-tuning.
  4. No agent code changes required. LiteLLM normalises the protocol.

Add a per-tenant budget override

  1. Set EAAP_AGENT__DAILY_USD_LIMIT as the tenant default.
  2. Set EAAP_AGENT__BUDGET_OVERRIDES as JSON: {"acme:billing-agent": {"daily_usd_limit": 50.00}}.
  3. Restart the app. The override applies on the next BudgetService.check() for that pair.
  4. Verify with the audit dashboard — you should see budget.checked events with the new limit applied.

Wire an MCP server into an agent

  1. Decide on transport: "stdio" for a local subprocess, "http" for a network server.
  2. Add an MCPServerSpec to your agent's mcp_servers() with a unique component_id.
  3. Run the agent. The SDK pools connections; tools surface to the LLM automatically.
  4. If two MCP servers expose tools with the same name, rename upstream — the resolver fails fast on collisions.

Write a unit test for an agent

  1. In conftest.py, build a container with AgentModule() + FakeModule().
  2. Override ILLMClient with a stub that returns canned LLMResponses.
  3. Override IPolicyEvaluator with FakePolicyEvaluator(default_allow=True).
  4. Construct the agent via container.get(YourAgent) and assert on its behaviour.
  5. Run with pytest -q. No Postgres, no OPA, no LLM keys needed.

Add a redactor for a new PII pattern

  1. This belongs to the platform team — the redactor chain is centrally maintained.
  2. Open a ticket describing the pattern (e.g. "passport numbers should be redacted in audit").
  3. Provide an example string the redactor should match. They'll add it; you don't need to change anything in your agent.
  4. If urgent, ping #eaap-platform-office-hours.
CH 11

Anti-patterns.

Six mistakes that look reasonable in week one and cause pain in week six. Block them in your own code review.

Reading os.environ directly in a tool

Settings are typed and validated. Reading the env directly bypasses validation, breaks dev/prod parity, and makes tests harder.

Use AppSettings from the container. Resolve secrets via SecretRef.
Catching Exception around an LLM call

You'll silently swallow BudgetExceededError and PolicyDenialError while looking like you're handling them.

Catch the specific SDK exception and return a structured error to the caller.
Skipping tenant_id in llm.complete()

Without it, budget enforcement quietly doesn't apply. Hits production. Hits the bill.

Pass tenant_id on every call. Use FakeBudgetService in tests.
Shipping a tool with opa_path=None

Defensible only for genuinely-no-policy cases. Most "I don't need a policy" claims are wrong on review.

Pick an opa_path; ask the platform team if the policy doesn't exist yet.
Logging customer data directly

Audit goes through a redactor chain. log.info(customer) doesn't.

Send durable records via AuditRecord; sanitise before any plain log.
Forking the SDK to "customise" something

You'll diverge from upgrades and lose the platform team's support. The DI container exists exactly so you don't have to fork.

Override the binding for the specific interface. Or ask the platform team to add the seam you need.
CH 12

Your first month.

A ramp plan. Adjust to your team's actual cadence — these are reasonable defaults, not orders.

Day 1

Hello, agent.

Install, scaffold, run the example agent. Read this guide front-to-back. Skim the engineer's handbook (CH 14).

Week 1

Your first tool.

Pick a small, real piece of business logic. Wrap it as a @tool. Add a unit test using fakes. Open a PR. Ask for a review from a senior on the team.

Week 2

Your first agent.

Subclass BaseAgent with 2–3 tools and a system prompt. Write a small eval set. Run it locally. Iterate on the prompt.

Week 3

Production-shaped.

Wire the agent behind FastAPI with OPAAuthorization. Add OPA policy with the platform team. Configure observability binds. Push to staging.

Week 4

Eval & ship.

Run the eval suite against staging. Compare to a baseline. If the numbers move in the right direction, ship a small slice to production. Watch the dashboards.

Month 2+

Steady state.

Iterate on prompts and tools, driven by eval results. Bring friction with the SDK to platform-team office hours. Resist the urge to fork.

CH 13

Asking for help.

Knowing where to ask saves hours. The platform team wants to hear from you — they hate finding out you struggled silently for a week.

CONFUSION ABOUT API
Read the engineer's handbook (docs/sdk-engineer-handbook.html) first. If it's not there, post in #eaap-platform-office-hours.
FEATURE GAP
Open a ticket describing the use case. If two teams ask for the same thing, the platform team will build it. Do not fork the SDK — it cuts you off from upgrades and from support.
PERFORMANCE ISSUE
If your agent is slow, first check observability dashboards. tenant_id + agent_id bind on every LLM call gives you a join key. If the SDK itself looks slow, ping platform.
POLICY FAILURE
Tools failing with PolicyDenialError? You probably need an OPA rule at your opa_path. Bring the path + the desired rule semantics to #eaap-security.
PRODUCTION INCIDENT
If the issue is in your agent code, you fix it. If it's in the SDK or platform infra, the on-call rotation is eaap-platform-oncall on PagerDuty.
PROMPT/EVAL DESIGN
Senior business AI engineers in your team are the right audience. Prompt engineering is craft — code review your prompts the way you code-review code.
office hoursThe platform team holds weekly office hours for the business AI team. Bring friction, gaps, and questions. It is the platform team's job to want to hear from you — they treat the friction list as input to their next sprint.
