A note on building agentic AI

Agents are non-deterministic. That single fact reorients everything.

Building agentic AI is closer to training a model than to wiring a workflow. It is not a thing you do in a drag-and-drop UI. It is a thing you do in code — and the cross-cutting concerns belong in one shared library.

§ 01Agents are non-deterministic. They behave like models, not like workflows.

An agent is not a flowchart with conditionals. It is a probabilistic system whose behaviour emerges from prompts, sampled tokens, retrieved context, tool outputs, and memory state. Same input on Monday and Tuesday produces different trajectories.

  • Same input, different output. Tokens are sampled from a distribution. The same user query routes through different tools, with different memory compaction, at different cost, on different days. A summarise_quarterly_report agent given the same 12-page PDF on two consecutive runs may call extract_tables three times in one run and once in the other, depending on what the model decides about its own confidence. The diff between the runs is a trace, not a config change.
  • Tool selection is decided at runtime by the model — not designed at build time by a human. This is the defining property that separates an agent from a workflow. A research agent given access to 40 tools picks 6 of them in a particular order based on tokens it just emitted. No flowchart can pre-draw that path; the path does not exist until the run happens.
  • Behaviour drifts when any input changes — model version, system prompt, tool catalog, retrieval index, temperature, even token-count thresholds. A regression in one component cascades into emergent behaviour elsewhere. Upgrading from one model snapshot to the next can quietly shift refusal rates by 8%. The fix is an evaluation suite, not a UI re-test.
  • The mature engineering response is the same one ML teams already use: golden-replay tests, evaluation harnesses, version-pinned dependencies, experiment tracking, observability over many runs.
An agent is much closer to a trained model than to a configured pipeline. We do not tune hyperparameters in a canvas. We do not build LLMs in a drag-and-drop UI. We should not build agents that way either.

§ 02Hyperparameters live in code. So do agent parameters.

No serious ML team has ever tuned a model in a drag-and-drop UI. The reason is structural, not cultural — and the same reasoning applies to agents.

  • Tuning is multi-dimensional. Adjusting temperature, top-p, retry count, refusal threshold, and tool budget together is a 5-D sweep. Code expresses this in a loop. A UI requires repetitive clicking with no audit of which combination was tested. optuna.create_study() sweeps 200 prompt-and-policy combinations overnight, logs every result to MLflow, and outputs a Pareto frontier of cost vs. accuracy. The UI equivalent is a spreadsheet of screenshots.
  • The interesting parameters are not parameters at all. They are functions: planning strategies, refusal classifiers, memory compaction policies, retry classifiers, output validators. Functions belong in code. "Re-plan if the JSON output's confidence field is below 0.7 AND the previous tool call took longer than 4 seconds AND the budget remaining is above $0.10." This is a conditional on a generated artifact. There is no UI primitive for it.
  • Experiment tracking is part of the build, not a separate tool. Code-first agents log every run to MLflow / W&B / LangFuse with the same ergonomics ML teams already use.
  • Reproducibility is non-negotiable. A regulator or incident investigator must be able to recreate the agent that ran on a given date. Code + lockfile + commit SHA does this. Vendor canvas state does not.

§ 03The agentic AI space is evolving faster than any UI can keep up with.

The set of techniques used to build robust agents in mid-2026 did not exist in the same form 18 months ago. The cadence is closer to ML research than to enterprise software.

  • New primitives ship quarterly. Constitutional AI / self-critique loops, agentic RAG, reflection and tree-of-thought variants, structured outputs, prompt caching, computer use, sub-agent orchestration, file-system agents, multi-modal tool returns. When prompt caching was released, code-first SDKs absorbed it via one decorator and shipped the same week. UI-first builders had to design new property panels, redo their docs, and roll out a vendor release. Some still do not expose it.
  • The Model Context Protocol (a standard for tool servers) was only finalised in late 2024. Code-first SDKs added MCP as a transport in days. UI builders had to redesign their tool-registration flow.
  • Memory architectures keep changing. Episodic vs. semantic vs. working memory, compaction with essential-entity preservation, retrieval over conversation history, summarisation tuned to entity boundaries. Each is a code primitive, not a checkbox.
  • Cost-aware planning is now a build-time concern. Model cascades (Haiku → Sonnet → Opus), per-tenant budget guards, refund-on-failure semantics — all need code to express, not configuration.
  • Output guardrails arrived after input guardrails. Detecting prompt-injection echoes, jailbreak responses, and PII leakage in model output is a post-generation filter. UIs that only check inputs missed this entire class of failure.
  • The publication cadence outruns the UI release cadence. A code-first SDK absorbs a new technique as an import. A UI builder has to design, ship, and document each one. The structural delay is real, and it costs.

§ 04What a drag-and-drop UI literally cannot represent.

Some agent behaviour is not "hard to draw" — it is unrepresentable in a node-and-edge metaphor. These are not edge cases; they are the substance of useful agents.

  • Conditional logic that depends on the model's own reasoning. "If the LLM's confidence-tag is below threshold, re-plan; if above, proceed." The condition lives on a generated string, not on a wire between two boxes.
  • Recursive sub-agent invocation. A research agent that spawns sub-agents per citation, depth bounded at runtime. Drawing this as a graph requires an unbounded canvas. A "deep research" agent finds 14 citations, spawns 14 sub-agents in parallel, each of which finds further citations. The shape of the call tree depends on what the model decides at runtime — there is nothing to draw at design time.
  • Memory compaction with essential-entity preservation. "When prompt tokens exceed 8000, summarise the conversation but preserve every key in essential_entity_keys (account_id, jurisdiction, currency)." This is conditional state mutation. UIs have nowhere to put it.
  • Tool selection driven by retrieval. "From the 200 available tools, give the model only the 8 nearest semantic neighbours of the user query." This is a runtime computation, not a designed graph.
  • Differential retry policies per error class. 429 → exponential backoff with jitter; 5xx → linear retry; 4xx → propagate; budget exceeded → refuse and re-plan. Each policy a code primitive, not a checkbox. Tenacity's retry_if_exception_type(...) classifies errors at the function level. The UI alternative is a giant table of error codes and dropdown actions, which still cannot express "retry once, then fall back to a cheaper model."
  • Tests that assert refusal. "Given a prompt-injection attempt, assert the agent did not call delete_account and emitted a structured refusal." This is a unit test. It does not exist in a canvas.
  • Streaming with mid-stream cancellation. "Cancel the stream if the partial output contains a banned phrase OR cumulative tokens exceed budget." Token-by-token logic. There is no UI primitive for it.

§ 05Cross-cutting concerns belong in one shared SDK — not eight team-by-team UI configurations.

The case for centralising in code is not aesthetic. It is operational. Every concern below is a bug-surface that multiplies if each agent reimplements it, and shrinks to one if a shared library handles it.

  1. Observability. OpenTelemetry spans for every LLM call, tool call, policy decision, and memory compaction — wired once, inherited by every agent. A new agent shipped on Tuesday gets full distributed tracing without anyone writing a single OTel call. The trace_id flows from the FastAPI request through the LLM call, the OPA decision, the tool invocation, the database query.
  2. Audit & redaction. Configurable redaction profiles, structured payloads, multiple sinks. PII never leaks because no team rolls their own. A regulated agent inherits the same redaction profile (regex + key-name + chain) as every other agent. There is no possibility of one team forgetting to redact a payment card number.
  3. Policy enforcement. OPA evaluation on every tool call, fail-closed on errors. Decisions logged and traced. A new tool registered by a junior data scientist is automatically subject to the same authorisation rules as a tool written by a principal engineer. The policy file is in git, reviewed in PR.
  4. Budget & cost attribution. Token + USD limits per tenant, agent, session — enforced before LLM calls, recorded after. A runaway agent loop is capped at $0.50 per session by a single configuration value, not by a custom guard each team writes (and inevitably forgets).
  5. Schema & contract management. Pydantic-typed tool I/O, versioned by (name, version), exported as JSON Schema for cross-language clients. A Python agent and a Java consumer agree on OrderLookupOut.v2. Deprecation of v1 is a structured event, not a Confluence page.
  6. Retry & resilience. Tenacity-classified retries (429 / 5xx / timeout); deterministic refusal of auth, validation, and budget errors. Same semantics across every agent.
  7. Testing seam. DI bindings allow swapping every external dependency for a fake. Unit tests run in milliseconds, no live infrastructure needed. A unit test for a loan_decision agent runs in 80ms with no Postgres, no OPA server, no LLM call — and still validates the full policy + budget + audit path.
  8. Vendor portability. Every concrete sits behind an interface. Swapping LiteLLM for Bedrock, OPA for Cedar, Postgres for DynamoDB is a one-line DI change. Agent code stays unchanged.

These concerns are eight bug-surfaces if eight teams reimplement them in eight UI configurations. They are one bug-surface, fixed once if centralised in a shared SDK.

§ 06A simple test: what does the artefact look like?

Set aside every philosophical argument. Ask one question: when this agent runs in production and something goes wrong at 3am, what does the on-call engineer open?

  • Code-first artefact. A Python package, version-pinned in a lockfile, reviewable in a PR, reproducible from a commit SHA, testable in CI without a live cluster, observable in OpenTelemetry, deployable like any other service.
  • UI-first artefact. A vendor-defined JSON blob — exportable but not reviewable, runnable only on the vendor's runtime, untestable without that runtime, observable only through the vendor's lens, deployable only through the vendor's pipeline.
  • The test: a regulator, an auditor, an SRE on call at 3am — three people who have never seen this agent before — must each be able to read the artefact, understand what it does, and reason about why it failed. The first artefact passes this test on day one. The second never does.

§ 07Where a UI does belong in agentic AI.

The argument is not that UIs have no place in agent systems. It is that UIs belong on top of code, not in place of it.

  • Operations consoles. Replay viewers, span explorers, budget dashboards, audit-log browsers, policy-override consoles for incident response. Click-driven workflows over a code-defined system. Indispensable, and obviously a UI.
  • Catalogue surfaces. Discovery pages where business users browse available agents, request access, view past results, trigger pre-built runs. Read-mostly, backed by the SDK's schema export.
  • Approval & governance flows. Human-in-the-loop checkpoints where a reviewer approves or rejects an agent action. The action is defined in code; the approval is a UI moment.
  • What does not belong in a UI: the agent itself. The decisions about which tools to call, how to compact memory, when to retry, when to refuse, how to validate output. Those are code.

The synthesis. Agents are non-deterministic systems built on a continuously evolving stack of techniques, validated through evaluation suites, with cross-cutting concerns that multiply per team if not centralised. The right tool to build them is the same tool every other non-deterministic, continuously-evolving system uses: code, with a shared library that absorbs the cross-cutting concerns once. UIs sit on top — for operators, reviewers, and consumers. The agent itself is a repository.
