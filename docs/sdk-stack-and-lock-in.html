The stack,
and what it won't
lock you into.

Every dependency is a future migration. This brief lists the libraries we picked, why, what the SDK abstracts behind an interface, and the realistic cost of swapping each one when the world changes.

The premise: the SDK is the dampening layer between your agents and the rest of the Python ecosystem. When LangGraph 0.0.x → 0.1, when LiteLLM ships a new provider, when Pydantic moves to v3, when OPA gets replaced by Cedar — the SDK absorbs the diff so your agents don't have to.

This document exists so that the answer to "what about lock-in?" is structured, honest, and verifiable against the source. We don't claim zero lock-in. We claim that lock-in is concentrated in places where it's worth paying for, and absent everywhere else.

§ 01

Three principles, applied everywhere.

The SDK's stance on dependencies follows three principles. They explain every binding choice, every interface, and every absence of an interface, in this document.

PRINCIPLE 01

Depend on interfaces, bind implementations

Every load-bearing dependency sits behind an I* interface. Agents call the interface; the DI container decides which implementation answers. Replace the implementation, keep the agents.

PRINCIPLE 02

Absorb library churn at the SDK boundary

When a dependency releases a breaking version, the SDK migrates internally — once — and ships a new SDK version. Agent code stays still. Versions are pinned at the SDK; agents pin only the SDK.

PRINCIPLE 03

Be honest about what's coupled

A handful of dependencies are deeply integrated: their replacement is a project, not a config flip. The SDK names them explicitly so you can plan for them rather than discover them mid-migration.

§ 02

The stack, as ports and adapters.

The SDK is the inner ring. Around it, an interface layer. Around that, the concrete dependencies that fulfil each interface. Around those, the external systems they talk to. Anything in the inner two rings is the SDK's contract; anything in the outer rings is replaceable.

§ 03

Stack inspection, dependency by dependency.

Each card has the same five fields: What it does, Why we picked it, SDK abstraction, Alternatives if we swap, and Evolution. The badge at top-right is the swap-cost rating: LOW means a DI binding swap, MEDIUM means contained refactor, HIGH means a real migration project.

LiteLLM unified LLM provider client

TIER 1 SWAP · LOW

What it does

One client surface that speaks OpenAI, Anthropic, Vertex, Bedrock, Azure, Cohere, and ~100 other providers. Token counting, cost lookups, streaming normalisation, batch.

Why we picked it

Provider-neutral by design. Active maintenance. Cost tables we can use for budget enforcement. We pin litellm>=1.35 and let the SDK absorb the weekly model additions.

SDK abstraction

Behind ILLMClient at di.interfaces:168. Agents and tools never import LiteLLM directly; they receive an injected ILLMClient.

Alternatives if we swap

  • LOWOpenRouter / Portkeyanother router; rebind ILLMClient.
  • LOWVendor SDK directlyAnthropic SDK or OpenAI SDK; one adapter class.
  • LOWIn-house gatewayyour team's HTTP wrapper; same interface, different impl.

Evolution

LiteLLM ships breaking response-shape changes occasionally. The SDK pins a tested range, normalises responses inside LiteLLMClient, and ships a new SDK version when an upgrade is necessary. Agent code never sees the diff.

Open Policy Agent policy decision service

TIER 1 SWAP · LOW

What it does

External policy decision point. Tools and APIs send {user, action, resource, claims, request}; OPA returns {allow, obligations, reason}. Rego is the policy language.

Why we picked it

De-facto standard for decoupled authorisation. Policies live outside agent code. Hot-reloadable, auditable, testable independently. Fail-closed semantics by default in our wrapper.

SDK abstraction

Behind IPolicyEvaluator at di.interfaces:262. OPAPolicyEvaluator is the production binding; NoOpPolicyEvaluator is the dev-only default.

Alternatives if we swap

  • LOWAWS Cedarswap evaluator class; rewrite policies in Cedar.
  • LOWCasbinsimilar shape, in-process; rebind.
  • LOWIn-house RBACsimple cases; rebind to your evaluator.

Evolution

OPA REST API is stable; the bigger churn is in your policies. The SDK gives you a typed decision shape so policy migrations don't ripple into agent code.

OpenTelemetry & LangFuse tracing + LLM observability

TIER 1 SWAP · LOW

What it does

OTel spans give you generic tracing across agents, tools, and HTTP. LangFuse layers on LLM-specific generations (prompt, completion, model, tokens, cost) for offline review and evals.

Why we picked it

OTel is the open standard; backends are interchangeable (Jaeger, Tempo, Datadog APM, Honeycomb). LangFuse is currently the best open offering for LLM-specific observability. Both fail open — backend errors don't break the agent.

SDK abstraction

Behind IObservabilityProvider at di.interfaces:88. RealObservabilityProvider is one binding; agents see only SpanContext.

Alternatives if we swap

  • LOWOTel + Heliconedifferent LLM-obs backend; same OTel.
  • LOWOTel onlydrop LangFuse; spans + logs are enough for many shops.
  • LOWDatadog APM onlyone provider for everything; rebind.
  • LOWIn-house tracerprint spans to a file; the interface doesn't care.

Evolution

LangFuse is the youngest dependency in this list and the most likely to be replaced. The interface boundary is exactly designed for that case — drop in a new provider class without touching agents.

PyJWT JWT verification

TIER 1 SWAP · LOW

What it does

Decodes and verifies signed JWTs. With the [crypto] extra it covers RSA/EC/EdDSA in addition to HMAC. Used at the FastAPI dependency layer.

Why we picked it

Stable, audited, pure-Python. Boring, in the good way. Wrapped behind our own JWTVerifier abstract base class so swapping it is mechanical.

SDK abstraction

JWTVerifier in security.jwt with HS256JWTVerifier and UnverifiedJWTDecoder shipped. Bind your own class for asymmetric keys, JWKS rotation, custom audience checks.

Alternatives if we swap

  • LOWauthlibJWKS rotation built-in; one new verifier class.
  • LOWjwcryptostrong crypto; swap if needed for FIPS.
  • LOWUpstream gatewayuse UnverifiedJWTDecoder; let your gateway sign-check.

Evolution

JWT itself is a frozen RFC. Library churn is minor. Replacement happens for compliance reasons (FIPS, custom HSMs) — the abstract base class makes that one file of code.

SQLAlchemy + asyncpg + Alembic persistence stack

TIER 2 SWAP · MEDIUM

What it does

Async ORM (SQLAlchemy 2.x), async Postgres driver (asyncpg), and migration tooling (Alembic). Used for budget state, audit-receipt tables, vector store, LangGraph checkpoints.

Why we picked it

Mature, async-native, type-friendly. asyncpg is the fastest Postgres driver. Alembic gives us reproducible schema migrations.

SDK abstraction

Behind IStorageProvider and ICheckpointSaver at di.interfaces:25, :278. Most agent code never imports SQLAlchemy directly.

Alternatives if we swap

  • MEDIUMTortoise / SQLModel / asyncpg-onlyschemas to port; interface stays.
  • MEDIUMDynamoDB / Mongodifferent model; new IStorageProvider impl + checkpointer rewrite.
  • LOWSQLite for devsame SQLAlchemy core; just a URL change.

Evolution

SQLAlchemy 2.x is committed long-term. Driver changes (asyncpg → psycopg-async) are URL-level. Pinning is conservative; we don't follow minor releases blindly.

fastmcp MCP client/server library

TIER 1 SWAP · LOW

What it does

Python implementation of the Model Context Protocol — a JSON-RPC dialect for tools, resources, and prompts. We use the client side to talk to MCP servers, with a connection pool, idle TTL, and health checks added on top.

Why we picked it

Reference implementation. Python-native. Active maintenance. The MCP standard itself is open — we're not betting on a vendor.

SDK abstraction

Behind IMCPConnectionFactory at mcp.transports:67. The agent declares an MCPServerSpec; the factory returns a session.

Alternatives if we swap

  • LOWAnthropic mcp-sdkanother Python MCP client; one factory.
  • LOWCustom JSON-RPC clientif you only need stdio.
  • LOWNo MCP at alldon't declare any servers; the factory is unused.

Evolution

MCP is a young protocol. The spec is evolving. The SDK pins a tested version of fastmcp and migrates centrally when the protocol moves; agents pin only the SDK.

structlog structured logging

TIER 2 SWAP · MEDIUM

What it does

Structured logs with task-isolated context (bind_context()), processor pipeline, and a JSON output formatter. Logs join with traces via trace_id.

Why we picked it

Best Python option for context-aware structured logging. Plays well with stdlib logging and OTel.

SDK abstraction

Indirect: get_logger() and bind_context() are the public surface. structlog is an implementation detail of those helpers — most agent code never imports it.

Alternatives if we swap

  • MEDIUMstdlib logging + ContextVarmore code in the helpers; same call sites.
  • MEDIUMlogurudifferent processor model; one helper rewrite.
  • LOWJust JSON to stdoutfor simple deployments; trivial helper.

Evolution

structlog is mature; major rewrites are unlikely. If JSON-log shapes change in your platform (e.g. ECS vs OTel logs), the change lives in one processor configuration, not in agent code.

Tenacity retry library (internal)

TIER 1 SWAP · LOW

What it does

Configurable retry policies with jitter, backoff, status-aware re-entry, and timeout-aware cancellation. Used inside LiteLLMClient and HTTP wrappers.

Why we picked it

The most flexible retry library in Python. We use it once, internally, behind well-tested defaults. Agent code never imports it.

SDK abstraction

Internal-only. Retry is a property of ILLMClient and the HTTP layer, not a concern callers configure.

Alternatives if we swap

  • LOWbackoffsimpler API; same call sites.
  • LOWHand-rolled20 lines for the simple cases.

Evolution

Tenacity has been API-stable for years. Replacement is unlikely; if it happens, it's one internal swap.

LangGraph agent orchestration framework

TIER 3 SWAP · HIGH

What it does

State-machine graph for agent loops. Provides the StateGraph, the add_messages reducer, checkpointing semantics, and node/edge primitives the SDK builds BaseAgent on top of.

Why we picked it

Most production-ready agent framework with explicit state, deterministic graphs, and a real checkpointer story. Pre-1.0 but actively maintained, with a clear path to stability.

SDK abstraction

Partial. BaseAgent.compile() and extend_graph() hide most of LangGraph from agent authors. But BaseAgent itself is built on StateGraph; replacement means rewriting BaseAgent internals.

Alternatives if we swap

  • HIGHCrewAI / AutoGendifferent state model; full BaseAgent rewrite.
  • HIGHCustom orchestrationasyncio + state machine; reasonable for small surface, big for ours.
  • MEDIUMDifferent LangGraph versionupgrade absorbed inside the SDK.

Evolution

This is the dependency we're most exposed to. The SDK's plan: stay on LangGraph through the 0.x → 1.0 transition (likely the largest single migration), watch for traction in alternative frameworks, and re-evaluate at 1.0. If we ever swap, agent code that uses BaseAgent + @tool stays largely unchanged — only the SDK internals move.

Pydantic + pydantic-settings type system & config

TIER 3 SWAP · HIGH

What it does

Runtime validation, schema generation, JSON-schema export, and typed settings from environment variables. The SDK uses Pydantic models everywhere — tool I/O, settings groups, audit records.

Why we picked it

De-facto standard for typed Python. Pydantic 2 is fast (Rust core), well-maintained, and integrates with FastAPI, OpenAPI, and JSON schema. Type checkers understand it.

SDK abstraction

None. Pydantic is part of the SDK's public surface — every tool defines Pydantic input/output, every settings group is a Pydantic model. We do not abstract it because we'd be reinventing it.

Alternatives if we swap

  • HIGHattrs + cattrsdifferent validation model; every tool I/O class touched.
  • HIGHmsgspecfaster, narrower; full migration of schemas.
  • MEDIUMPydantic v3 (when it lands)SDK absorbs; agent code adjusted only where v3 itself is breaking.

Evolution

The Pydantic 1 → 2 migration was painful for the ecosystem. v3, when it ships, will be smaller. The SDK pins pydantic>=2.6 and tracks minor releases. Replacement is not currently on the table.

injector dependency injection

TIER 2 SWAP · MEDIUM

What it does

Provides the @inject decorator and the binding mechanism that lets BaseAgent declare its dependencies as constructor arguments. The SDK wraps it in a Container facade.

Why we picked it

Small, focused, no surprises. Sufficient for the SDK's needs. We don't expose it directly — agents see Container only.

SDK abstraction

Wrapped by Container at di.container. Agents call Container.build([modules]), container.get(I), container.override(I, fake) — never injector directly.

Alternatives if we swap

  • MEDIUMdependency-injectordifferent DSL; rewrite Container internals.
  • MEDIUMpunqsimpler container; rewrite Container internals.
  • MEDIUMHand-rolled~200 lines; we'd consider this if injector ever stalled.

Evolution

injector has had no breaking changes in years. The SDK could replace it with a hand-rolled container in a week if the project ever went unmaintained.

FastAPI HTTP layer (optional)

OPTIONAL SWAP · OPTIONAL

What it does

Async web framework. We provide OPAAuthorization as a FastAPI Depends() dependency for teams who want to expose agents over HTTP with JWT + OPA enforcement.

Why we picked it

Most popular async-first Python framework. Plays well with Pydantic. Most teams will already have it. Including it as a default integration saves time.

SDK abstraction

Not an interface — it's an integration. Agents and tools have no FastAPI dependency. Use it if you want HTTP; ignore it otherwise.

Alternatives if we swap

  • LOWLitestar / Starlette / aiohttproll your own dependency; the SDK doesn't care.
  • LOWNo HTTP at allworkers, batch jobs, event-driven — agents don't need a web framework.

Evolution

FastAPI is on a stable major version. Starlette underneath ships with predictable cadence. If you're worried about coupling, write your own dependency function — it's about 30 lines.

typer + rich + Jinja2 CLI scaffolding (optional)

OPTIONAL SWAP · OPTIONAL

What it does

The eaap CLI: project scaffolding, settings inspection, dev probes. Typer for command parsing, Rich for output, Jinja2 for templates.

Why we picked it

Standard, boring choices for a Python CLI. None of these touch agent runtime — they exist for ergonomics during development.

SDK abstraction

Lives entirely in ai_core.cli. Agents and tools don't import it. Drop the CLI extra and the SDK still works at runtime.

Alternatives if we swap

  • LOWClick instead of Typersame author family; one rewrite.
  • LOWNo CLI at allyour team writes its own scaffolds.

Evolution

Mature stack. No expected churn. Replacement is convenience, not necessity.

Sentry · Datadog · AWS extras opt-in integrations

OPTIONAL EXTRAS SWAP · OPTIONAL

What it does

SentryAuditSink and DatadogAuditSink ship audit events to those platforms. storage-aws and secrets-aws extras add AWS S3 / Secrets Manager backends.

Why we picked them

Most enterprises run one of these; opt-in extras let teams pull only what they need. Imports are guarded — missing extras raise ConfigurationError at the boundary, not deep in the call.

SDK abstraction

Audit sinks are bound through IAuditSink; storage providers through IStorageProvider; secrets through ISecretManager. Add or remove extras without touching agent code.

Alternatives if we swap

  • LOWGCP / Azure equivalentswrite a GCSStorageProvider / VaultSecretManager; bind it.
  • LOWIn-house equivalentssame pattern; trivial.

Evolution

Vendor SDKs change frequently; the SDK pins each extra independently and isolates the breakage to its sink/provider class.

honest about lock-in

Two dependencies are deeply coupled to the SDK and would be a real migration to replace: LangGraph (agent orchestration) and Pydantic (the type system). The rest sit behind interfaces and can be swapped with minimal-to-moderate effort.

If LangGraph or Pydantic ever needed replacing, the SDK would absorb most of the pain on behalf of agent teams. The point isn't zero lock-in — every system has some — the point is putting the lock-in where it costs the least to maintain.

§ 05

Evolution, by example.

The SDK is the dampening layer. When dependencies move, the SDK absorbs the diff so agents don't have to. Here's what that looks like in practice.

2024 · LiteLLM 1.32 → 1.40

Streaming response shape changed

LiteLLM updated the streaming chunk schema for tool calls. The diff lived in LiteLLMClient._normalize_response. Agent code: zero changes. Tools that consumed LLMResponse kept the same shape on both sides of the upgrade.

2024 · LangGraph 0.0.50 → 0.0.55

Reducer signature broadened

add_messages gained the REMOVE_ALL_MESSAGES marker for compaction. The SDK adopted it inside MemoryManager._do_compact. Agent code: zero changes; the compaction node itself is internal to BaseAgent.

2024 · Pydantic v2 minor releases

Validator decorator semantics tightened

Pydantic 2.6 → 2.8 sharpened error messages on field_validator. The SDK pinned the version, ran the test suite, and shipped. Agent tools using Pydantic: zero changes; new error messages improved diagnostics for free.

Future · LangGraph 1.0

Anticipated graph-API stabilisation

The 0.x → 1.0 transition will rename a handful of internal graph constructs. The SDK's plan: stage the migration in a feature branch, run the full test matrix on the new version, ship a single SDK release that absorbs the diff. Agents using BaseAgent + @tool: expected zero changes.

Future · Pydantic v3

Possible Rust-core re-architecture

v3 is on the roadmap but not imminent. When it arrives, the SDK pins to v2 until v3 is stable, then migrates schemas centrally. Tools that follow the SDK's idiomatic Pydantic usage will need a small re-validate; tools that reach for v2-specific internals will need attention. The SDK's CHANGELOG will name the affected APIs explicitly.

The evolution promise, in three rules

  1. The SDK pins major versions; agents pin only the SDK.
  2. Breaking changes in dependencies are absorbed inside the SDK, with the affected public APIs (if any) called out in the CHANGELOG.
  3. Deprecations get a release-cycle of warning before removal; you'll never wake up to a broken build because we silently dropped something.
§ 06

Worked example: swapping OPA for Cedar.

A concrete walk-through to make "swap is low cost" verifiable rather than rhetorical. The same pattern applies to every Tier-1 dependency.

Replace OPA with AWS Cedar

SCENARIO · COMPLIANCE TEAM MOVES TO CEDAR · 1–3 DAYS OF WORK

Cedar is AWS's policy language and evaluator. The wire shape differs from OPA, but the SDK's IPolicyEvaluator doesn't care — it asks for a typed PolicyDecision back; how you get there is up to you.

  1. Add cedarpy (or your Cedar binding of choice) to requirements.txt. This is the only new dependency.
  2. Write CedarPolicyEvaluator(IPolicyEvaluator): a class with an async evaluate(*, decision_path, input) -> PolicyDecision method that calls Cedar and translates the result.
  3. Bind your evaluator in a custom DI module: binder.bind(IPolicyEvaluator, to=CedarPolicyEvaluator, scope=singleton).
  4. Replace ProductionSecurityModule() with CedarSecurityModule() in your app bootstrap. No agent code changes; no tool code changes; no FastAPI handler changes.
  5. Port your Rego policies to Cedar (this is the actual work; the SDK doesn't help here, but it doesn't hurt either).
  6. Run the full test suite. The SDK's contract tests for IPolicyEvaluator verify your new class behaves correctly. Fakes (FakePolicyEvaluator) are unaffected.
  7. Deploy. Audit events automatically include the new decision_path values.

What didn't change: every @tool, every BaseAgent subclass, every FastAPI route, every test that uses FakePolicyEvaluator. The lock-in lived in one binding.

The same shape applies to swapping LiteLLM for OpenRouter, OTel for Datadog APM, fastmcp for the Anthropic MCP SDK, PyJWT for authlib, EnvSecretManager for VaultSecretManager. One adapter class per dependency. One DI binding to flip. Agent code stays still.

Lock-in is a design choice,
not a fact of life.

Every dependency in the SDK is there because the alternative is reinventing it. Every load-bearing dependency sits behind an interface because the alternative is being unable to swap it. And every public API the SDK exposes is documented so that when something does change, you know exactly what's changing and why.

Bring lock-in concerns to the platform team. If a dependency feels too coupled, the right answer is usually a new interface, not a fork.
