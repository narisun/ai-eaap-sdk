## 1. Architecture

Phase 13 ships two narrow, additive features that build directly on Phase 4's solid budget/latency foundations — both default-disabled so existing deployments see zero behavior change.

### Two deliverables

1. **LLM latency SLO (alert-only).** New optional `LLMSettings.latency_slo_ms: int | None = None` field. When set, after each LLM call returns, `LiteLLMClient` compares wall-time `latency_ms` against the threshold and — on violation — emits an observability event `llm.slo_violated` with `model`, `agent_id`, `tenant_id`, `latency_ms`, `threshold_ms`. The response is unchanged. No new exception, no flow control. Operators build alerts from the event (existing Sentry, Datadog, OTel sinks already receive `IObservabilityProvider.record_event` calls).

2. **Per-agent budget overrides (settings-driven).** New `BudgetOverride` Pydantic model carrying optional `tenant_id`, `agent_id`, `daily_token_limit`, `daily_usd_limit`. New `BudgetSettings.overrides: list[BudgetOverride] = []` field. `InMemoryBudgetService` gains a small per-field resolver: for each limit, walk overrides from most-specific `(tenant, agent)` → tenant-only `(tenant, None)` → agent-only `(None, agent)` → global default; take the first non-None value. Per-field resolution lets a partial override compose with defaults cleanly.

### Module layout

```
src/ai_core/config/settings.py
  - BudgetOverride (new BaseModel)
  - BudgetSettings — add overrides field
  - LLMSettings — add latency_slo_ms field

src/ai_core/llm/budget.py
  - InMemoryBudgetService — add _resolve_limits(tenant, agent) → (token, usd);
    check() consults it instead of reading defaults directly.

src/ai_core/llm/litellm_client.py
  - complete() — add post-call SLO check + record_event call (alert-only).

tests/unit/llm/test_budget.py
  - 7 new tests for override resolution (exact match, tenant-only, agent-only,
    most-specific-wins, partial overrides, no-override fallback, duplicate handling).

tests/unit/llm/test_litellm_client.py
  - 4 new tests for SLO (disabled, within, violated with attrs, exact threshold).

docs/settings.md
  - Regenerated by Phase 10's auto-generator (drift test gates it).
```

### Out of scope (deferred)

- **Tool / MCP latency SLO.** v1 is LLM-only. The infrastructure for tool-call latency capture exists (`ToolInvoker` records `latency_ms` per call); a follow-up phase can layer the same SLO check there.
- **Hard-enforce mode (`raise` on violation).** A future `latency_enforcement: Literal["alert_only", "raise"]` setting is a clean extension; alert-only is v1.
- **Granular time windows on budgets.** Hourly / weekly / monthly limits remain deferred. v1 keeps the existing daily reset.
- **Runtime override registration API.** Operators wanting dynamic limits subclass `InMemoryBudgetService` and override `check()` — the existing escape hatch.
- **Per-tool / per-model SLOs.** v1 is a single global threshold for all LLM calls.
- **Dashboards / alert rules.** The SDK emits the event; teams compose dashboards from their existing OTel / Sentry / Datadog setups.

### Backward compatibility

- `latency_slo_ms` default `None` → no SLO check runs → identical behavior to today.
- `overrides` default `[]` → resolver always falls through to global defaults → identical behavior to today.
- All existing budget tests continue to pass without modification (they don't set overrides).
- All existing LLM client tests continue to pass without modification (they don't set `latency_slo_ms`).
- `IBudgetService` interface signature unchanged. `InMemoryBudgetService.__init__` signature unchanged. The new `_resolve_limits` is a private method on the existing class.

## 2. Components & data flow

### Component 1 — `BudgetOverride` (`src/ai_core/config/settings.py`)

New Pydantic model placed adjacent to `BudgetSettings`:

```python
class BudgetOverride(BaseModel):
    """Per-(tenant, agent) budget override entry.

    Resolution by InMemoryBudgetService is per-field, most-specific-wins:
    1. (tenant_id, agent_id) — both set → exact pair
    2. (tenant_id, None) — tenant-only — matches all agents under tenant
    3. (None, agent_id) — agent-only — matches that agent across all tenants

    Each limit field is optional; partial overrides compose with defaults
    (e.g. an entry that only sets daily_token_limit leaves daily_usd_limit
    falling through to the global default).

    An entry with both tenant_id=None AND agent_id=None never matches anything
    (the resolver does not include the all-None candidate). Such entries are
    silently inert — no validation error at load time, but no effect either.
    """

    model_config = ConfigDict(extra="ignore")

    tenant_id: str | None = None
    agent_id: str | None = None
    daily_token_limit: int | None = Field(default=None, ge=0)
    daily_usd_limit: float | None = Field(default=None, ge=0.0)
```

### Component 2 — `BudgetSettings` extension

### Component 4 — `InMemoryBudgetService._resolve_limits` (`src/ai_core/llm/budget.py`)

```python
def _resolve_limits(
    self,
    tenant_id: str | None,
    agent_id: str | None,
) -> tuple[int, float]:
    """Resolve effective (daily_token_limit, daily_usd_limit) for a key.

    Walks override candidates from most-specific to least-specific:
    (tenant, agent), (tenant, None), (None, agent). For each candidate,
    finds matching override entries and fills in any field still unset.
    Falls back to settings defaults for any field not covered by overrides.

    Resolution iterates `self._settings.overrides` in list order; first match
    wins per-field. Operators wanting deterministic precedence put more
    specific entries first.
    """
    candidates: list[tuple[str | None, str | None]] = []
    if tenant_id is not None and agent_id is not None:
        candidates.append((tenant_id, agent_id))
    if tenant_id is not None:
        candidates.append((tenant_id, None))
    if agent_id is not None:
        candidates.append((None, agent_id))

    token: int | None = None
    usd: float | None = None
    for cand_tenant, cand_agent in candidates:
        for ov in self._settings.overrides:
            if ov.tenant_id == cand_tenant and ov.agent_id == cand_agent:
                if token is None and ov.daily_token_limit is not None:
                    token = ov.daily_token_limit
                if usd is None and ov.daily_usd_limit is not None:
                    usd = ov.daily_usd_limit
        if token is not None and usd is not None:
            break

    return (
        token if token is not None else self._settings.default_daily_token_limit,
        usd if usd is not None else self._settings.default_daily_usd_limit,
    )
```

The existing `check()` body replaces direct reads of `default_daily_token_limit` / `default_daily_usd_limit` with `token_limit, usd_limit = self._resolve_limits(tenant_id, agent_id)`.

### Component 5 — `LiteLLMClient` SLO check (`src/ai_core/llm/litellm_client.py`)

After the existing `record_llm_usage(latency_ms=..., ...)` call, add (inside the same try/except wrapper that already protects observability calls):

```python
slo_ms = self._settings.llm.latency_slo_ms
if slo_ms is not None and latency_ms > slo_ms:
    await self._observability.record_event(
        "llm.slo_violated",
        attributes={
            "llm.model": model_used,
            "llm.tenant_id": tenant_id or "",
            "llm.agent_id": agent_id or "",
            "llm.latency_ms": latency_ms,
            "llm.threshold_ms": slo_ms,
        },
    )
```

`record_event` is already part of `IObservabilityProvider`. Strict `>` semantics — exactly-on-threshold is not a violation. The check uses `model_used` (the resolved model name, including any fallback) so events tag the actual model that was slow.

### Data flow

**Budget check (with override):**

```
1. LiteLLMClient.complete(tenant_id="acme", agent_id="customer-support")
2. estimate_tokens → call IBudgetService.check(tenant_id, agent_id, est)
3. InMemoryBudgetService.check:
   a. _resolve_limits("acme", "customer-support") →
      - candidates: [("acme","customer-support"), ("acme",None), (None,"customer-support")]
      - finds override (tenant_id="acme", agent_id="customer-support",
        daily_token_limit=2000000, daily_usd_limit=None)
      - token=2000000 (from override); usd still None
      - tries (acme, None) — no match for usd field
      - tries (None, customer-support) — no match
      - usd falls back to default (50.0)
      - returns (2000000, 50.0)
   b. Compares projected spend to (2000000, 50.0); BudgetCheck.allowed=True
4. LLM call proceeds.
```

**SLO violation (alert-only):**

```
1. LiteLLMClient.complete(...) → 7.2s wall time
2. record_llm_usage(latency_ms=7200, ...) — existing flow, OTel + LangFuse
3. settings.llm.latency_slo_ms == 5000 → 7200 > 5000 → violation
4. record_event("llm.slo_violated", attributes={
       "llm.model": "claude-sonnet-4-6",
       "llm.tenant_id": "acme",
       "llm.agent_id": "customer-support",
       "llm.latency_ms": 7200.0,
       "llm.threshold_ms": 5000,
   })
5. Returns LLMResponse normally — caller sees no error.
6. Operator's OTel/Sentry/Datadog alert rule fires on the event name pattern.
```

## 3. Error handling, testing, constraints

### Error handling

- **`record_event` raises during SLO emission.** `IObservabilityProvider.record_event` already follows the SDK's "observability boundary" pattern: failures are logged but never propagate. The new SLO emission inherits that posture — if the OTel exporter is misconfigured, the LLM call still returns the response cleanly. The existing `try/except` wrapper in `LiteLLMClient` around `record_event` calls handles this; the new emission goes inside the same wrapper.

- **Misconfigured `BudgetOverride` entries.** Pydantic field validation handles negative limits at `AppSettings` load time (`Field(ge=0)`). A `BudgetOverride` with both `tenant_id=None` AND `agent_id=None` is a no-op (the resolver's candidate list never includes `(None, None)`), so silently dead-weight rather than an error — the resolver code documents this. We don't reject at load time because rejecting would be a footgun for partial-config workflows where overrides are populated by external tooling.

- **Multiple matching overrides for the same candidate.** Resolution iterates `self._settings.overrides` in list order; first match wins per-field. Documented in `_resolve_limits` docstring. Config order is semantically meaningful — operators put more specific entries first if both could match. Deterministic, predictable.

- **`latency_slo_ms = 0`.** Would trigger a violation on every LLM call (since `latency_ms > 0` is essentially always true after the first millisecond). Pydantic's `ge=0` allows `0` syntactically; if operators set this it's their call. Documented in the field description; no separate guard.

- **No new exception classes.** Alert-only SLO doesn't need an exception. Override resolution funnels into the existing `BudgetExceededError` raised by `InMemoryBudgetService` when the resolved limit is breached.

- **`BudgetCheck.reason` enrichment on denial** is a polish item the implementer may add (e.g., naming which override matched in the denial message). Not a hard requirement of this spec — the structured `BudgetCheck.reason` field is text-only and operators primarily diagnose via audit records that already carry tenant/agent. Leave to implementer judgment.

### Testing strategy

- **`tests/unit/llm/test_budget.py` extension** — 7 new tests for override resolution. All use `InMemoryBudgetService` directly with custom `BudgetSettings(overrides=[...])` — no DI plumbing needed, fast unit tests.
- **`tests/unit/llm/test_litellm_client.py` extension** — 4 new tests for SLO. Use the existing `FakeObservabilityProvider` (records `record_event` calls in its `events` list).
- **Backward compatibility.** All existing tests pass without modification:
  - Existing `test_budget.py` tests don't set `overrides` → list is empty → resolver returns global defaults — semantically identical to today's `default_daily_token_limit` / `default_daily_usd_limit` reads.
  - Existing LLM client tests don't set `latency_slo_ms` → check is skipped → no behavior difference.
  - `FakeBudgetService` (in `ai_core.testing.fakes`) is unchanged. Tests using it continue to pass.
- **Settings doc drift.** Phase 10's `tests/unit/config/test_settings_doc_drift.py` will fail after the new fields land until `docs/settings.md` is regenerated. The plan includes an explicit task step.
- **Smoke gate.** No changes — `scripts/run_examples.sh` doesn't exercise budget/SLO paths.

### Constraints

- **Default-disabled.** `latency_slo_ms = None` and `overrides = []` are the defaults. Adding Phase 13 to a deployment without setting either changes nothing.
- **No new runtime dependencies.** Pydantic v2 already imported. No new SDK modules. No FastMCP / observability / audit changes.
- **No new exception classes, no new error codes.** Alert-only SLO is a `record_event` call. Override resolution funnels into the existing `BudgetExceededError` raised by `InMemoryBudgetService` when the resolved limit is breached.
- **`BudgetOverride` is a `BaseModel`, not a `BaseSettings`.** It's a config sub-shape consumed by `BudgetSettings`, not a settings group. Lives in `settings.py` alongside `BudgetSettings`.
- **Override resolution is internal to `InMemoryBudgetService`.** The `IBudgetService` interface signature is unchanged. Subclasses that override `check()` for distributed deployments (Redis/DB-backed) decide their own resolution — they can call `self._resolve_limits(tenant, agent)` if they subclass `InMemoryBudgetService`, or implement entirely different logic if they subclass the interface directly.
- **Per-field resolution means no "complete override wins" semantics.** A user who wants a single override entry to fully replace defaults must set both `daily_token_limit` and `daily_usd_limit` on it. Documented in the `BudgetOverride` docstring. We picked per-field because partial overrides are common.
- **`latency_slo_ms` is global across all LLM calls.** Per-model thresholds are deferred — the v1 contract is a single threshold.
- **No SLO violation history or aggregation in the SDK.** The SDK emits one event per violation; teams aggregate in their observability stack.
- **Backward-compat for `InMemoryBudgetService.__init__`.** No signature change. The new `_resolve_limits` is a private method added to the existing class.
